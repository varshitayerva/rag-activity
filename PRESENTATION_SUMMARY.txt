┌─ WHAT IS M1? ──────────────────────────────────────────────────────────────────┐
│                                                                                 │
│  M1 takes raw documents and transforms them into optimized chunks for AI       │
│                                                                                 │
│  INPUT:  PDF or Markdown files                                                 │
│  PROCESSING:                                                                    │
│    1. Parse file (extract text)                                                │
│    2. Detect sections and structure                                            │
│    3. Split intelligently into chunks                                          │
│    4. Extract metadata (author, category, etc)                                 │
│    5. Store in PostgreSQL database                                             │
│  OUTPUT: Chunks with metadata ready for embeddings and retrieval               │
│                                                                                 │
└─────────────────────────────────────────────────────────────────────────────────┘

┌─ THE PROBLEM M1 SOLVES ────────────────────────────────────────────────────────┐
│                                                                                 │
│  NAIVE CHUNKING (Fixed 500-token windows):                                     │
│  ────────────────────────────────────────                                      │
│  Document: "## Step 1: Do this. ## Step 2: Do that. ## Step 3: Final step."   │
│                                                                                 │
│  ❌ RESULT:                                                                     │
│    Chunk 1: "## Step 1: Do this. ## Step 2: Do"                                │
│    Chunk 2: "that. ## Step 3: Final"                                           │
│    Chunk 3: "step."                                                            │
│                                                                                 │
│  Problem: Steps are fragmented! LLM can't see complete information.            │
│                                                                                 │
│  ──────────────────────────────────────────────────────────────────────────    │
│                                                                                 │
│  SEMANTIC CHUNKING (Intelligent boundaries):                                    │
│  ────────────────────────────────────────                                      │
│  Document: SAME FILE                                                           │
│                                                                                 │
│  ✅ RESULT:                                                                     │
│    Chunk 1: "## Step 1: Do this."                                              │
│    Chunk 2: "## Step 2: Do that."                                              │
│    Chunk 3: "## Step 3: Final step."                                           │
│                                                                                 │
│  Solution: Each step complete! LLM gets full context for better answers.       │
│                                                                                 │
│  IMPROVEMENT: 55% better accuracy (40% → 95%)                                  │
│                                                                                 │
└─────────────────────────────────────────────────────────────────────────────────┘

┌─ WHAT GETS STORED IN DATABASE ─────────────────────────────────────────────────┐
│                                                                                 │
│  DOCUMENTS TABLE (12 columns)                                                   │
│  ─────────────────────────────                                                 │
│    ✓ Document metadata: filename, type, department, category                  │
│    ✓ Statistics: page_count, total_tokens, chunks_created                     │
│    ✓ Strategy: which chunking method was used                                 │
│    ✓ Timestamp: when uploaded                                                 │
│    ✓ Metadata: custom JSON data                                               │
│                                                                                 │
│  CHUNKS TABLE (10 columns)                                                     │
│  ─────────────────────────                                                     │
│    ✓ Content: full text of chunk                                              │
│    ✓ Structure: section name, page number, chunk index                        │
│    ✓ Metrics: token count                                                     │
│    ✓ Reference: document_id (foreign key)                                     │
│    ✓ For future: embedding_vector (for M2)                                    │
│                                                                                 │
│  RELATIONSHIPS:                                                                 │
│  ──────────────                                                                │
│  One Document → Many Chunks (1 to N relationship)                             │
│                                                                                 │
│  Example:                                                                       │
│    Document: troubleshooting-guide.md                                          │

│  ──────────────────────                                                        │

│                                                                                 │
│  M1 → M4 (Caching)                                                             │
│  ─────────────────                                                             │
│  M4 wraps M1 output with embedding cache                                       │
│  Reduces API calls, faster retrieval                                           │
│  Improved performance for repeated queries                                      │
│                                                                                 │
│  M1 → M5 (Frontend)                                                            │
