PART 2: WHY WE CHOSE THIS APPROACH
================================================================================

PROBLEM #1: FAISS Not Persistent
─────────────────────────────────
Old: Vectors stored in-memory, lost on restart
Why It Failed: Had to re-embed all documents every time app restarted
Solution: pgvector stores in PostgreSQL (persistent)
Result: Embeddings survive restarts, no re-processing needed

PROBLEM #2: Pickle Serialization Inefficient
──────────────────────────────────────────────
Old: Embeddings pickled as binary blobs
Why It Failed: Wasteful (bloated), slow parsing
Solution: Native pgvector type (1536-dimensional)
Result: Smaller storage, faster queries

PROBLEM #3: Python-Side Similarity Searches O(n)
──────────────────────────────────────────────────
Old: Fetch all vectors, compare in Python (linear time)
Why It Failed: Slow, doesn't scale
Solution: HNSW index in PostgreSQL (O(log n))
Result: 4ms search vs seconds before

PROBLEM #4: Search Misses Results
──────────────────────────────────
Old: Vector search only
Why It Failed:
  - Q: "restart pod" vs "kubectl delete pod"
  - Vector: Might miss if different embeddings
  - Keyword: Catches exact words
Solution: Hybrid (vector + BM25)
Result: Better recall, catches semantic + keyword

PROBLEM #5: Ranking Results Poorly
───────────────────────────────────
Old: Just return vector similarity score
Why It Failed: Misses keyword matches, rank wrong
Solution: RRF fusion (combine vector + keyword ranks)
Result: Intelligent ranking of combined results

FINAL ARCHITECTURE:
┌─ PostgreSQL (single database)
│  ├─ pgvector (1536-dim embeddings)
│  ├─ HNSW index (O(log n) search)
│  └─ All tables (docs, chunks, metadata)
│
├─ BM25 (in-memory, rebuilt on startup)
│  ├─ Keyword indexing
│  └─ Full-text matching
│
└─ RRF Fusion
   ├─ Combine vector + BM25 scores
   └─ Intelligent ranking
